## 1. 框架定位与能力边界

`Known` 是一个基于 `Blazor` 的插件化业务开发框架，核心特征：

- **全栈 C#**：前后端统一语言与模型。
- **插件化组织**：按模块扩展，降低耦合。
- **内置通用能力**：权限、菜单、字典、安装、工作流、导入导出、代码生成。
- **多运行模式**：`Server`、`Wasm + Host`、`WinForm`、`MAUI`、`Photino`。

适合：中后台系统、低代码平台、业务中台、跨端管理系统。

---

## 2. 代码结构总览

结合当前仓库：

- `Known/`：核心类库（模型、组件、扩展、ORM、权限、UI 运行时）
- `Plugins/Known.Core/`：Web 后端能力（控制器、认证、动态 WebApi、中间件挂载）
- `Plugins/Known.Sample/`：示例业务插件（菜单、页面、服务、实体、流程）
- `Known.Server/`：Server 模式宿主示例
- `Known.Wasm/`：WebAssembly 客户端宿主示例
- `Known.WasmHost/`：Wasm 后端宿主示例
- `Known.Maui/`、`Known.WinForm/`、`Known.Photino/`：桌面/跨端宿主示例
- `Document/`：项目文档

### 目标框架

- 核心/插件库多目标：`net8.0;net9.0;net10.0`
- 示例宿主：当前以 `net10.0` 为主

## 3. 运行模式与启动链路

## 3.1 Server 模式

入口参考：`Known.Server/Program.cs`

关键流程：

1. `builder.Services.AddApplication(...)`
2. `app.UseApplication()`
3. `app.MapRazorComponents<App>()...`

`AddApplication` 在 `Known.Server/AppConfig.cs` 中实现，内部执行：

- `services.AddKnown(...)`：注册框架基础能力
- `services.AddSample()`：注册业务插件
- `services.AddKnownWeb(action)`：注册 Web 后端与认证、控制器等

## 3.2 Wasm + Host 模式

- 客户端：`Known.Wasm/Program.cs` 使用 `AddApplicationClient`
- Host：`Known.WasmHost/Program.cs` 使用 `AddApplicationWeb`

这套模式本质是：

- UI 在浏览器 WebAssembly 运行
- 后端 API/鉴权/静态资源由 Host 提供

## 3.3 Desktop（WinForm / MAUI / Photino）

通过 `services.AddKnownDesktop(...)` 复用核心能力，关闭部分 Web 特性（如通知 Hub）。

## 4. 配置体系（Config / CoreConfig / UIConfig）

## 4.1 `Config`（全局应用配置）

常用项：

- `Config.App`：应用基础信息（Id、Name、目录、数据库等）
- `Config.RenderMode`：渲染模式（Server/Auto）
- `Config.Assemblies`：模块程序集清单
- `Config.Modules`：一级模块定义
- `Config.AddModule(assembly)`：装配并触发扫描

## 4.2 `CoreConfig`（后端行为配置）

常用扩展点：

- `OnInitial` / `OnAdmin`
- `OnInstall` / `OnInstallModules`
- `OnRoleModule`
- `OnRegistering` / `OnRegistered`
- `OnLoging` / `OnLoged`
- `CheckSystem`（License 校验）

## 4.3 `UIConfig`（前端 UI 运行配置）

可控制：

- 系统标签页扩展（`SystemTabs`、`UserTabs`、`ModuleTabs`）
- 忽略鉴权路由（`IgnoreRoutes`）
- 错误页文案（`Errors`）
- 布局挂钩（`AppBody`、`AdminBody`）

## 5. 模块自动发现机制（约定优于配置）

核心在 `Known/Helpers/InitHelper.cs` + `Known/Helpers/CoreHelper.cs`。

程序集加入后，框架会扫描类型并按特性自动处理：

- `[Service]`：注册依赖注入
- `[Client]`：注册客户端代理
- `[WebApi]`：生成动态 API 元数据
- `[Menu]` / `[AppMenu]` / `[Role]`：菜单与权限元数据
- `[Import]`：导入器注册
- `[Task]`：任务注册
- 继承 `EntityBase`：实体模型注册
- 继承 `FlowBase`：流程处理类注册

这使得“新增模块”通常不需要改中心化注册表。

---

## 6. 页面开发模型（Blazor 代码式构建）

`Known` 采用代码构建 UI（`RenderTreeBuilder` + 组件模型）为主。

常见基类：

- `BasePage`：页面根基类
- `BaseTablePage<T>`：列表表格页面
- `BaseForm<T>`：表单页面
- `BaseTabPage` / `BaseStepPage`：标签页/步骤页

示例：`Plugins/Known.Sample/Pages/Produce/WorkList.cs`

典型模式：

1. `OnInitPageAsync` 初始化表格模型、Tab、Form 配置
2. `[Action]` 标注页面操作（新增/编辑/删除/打印）
3. `Table.OnQuery = ...` 绑定查询委托

---

## 7. 服务开发模型（Client + Service + WebApi）

推荐三段式（见 `ProduceService.cs`）：

1. 接口：`IProduceService : IService`
2. 客户端实现：`[Client] class ProduceClient : ClientBase`
3. 服务端实现：`[WebApi, Service] class ProduceService : ServiceBase`

优势：

- 接口契约单一来源
- Server 与 Wasm 共用 DTO 与签名
- 动态 API 自动挂载，减少手工 Controller

### 请求鉴权机制

`ClientBase.Http` 会自动写入 `Constants.KeyToken` 请求头；
服务端通过上下文恢复 `CurrentUser`。

## 8. 数据访问与实体模型

## 8.1 实体基类

- `EntityBase<TKey>` / `EntityBase`
- 默认包含审计字段（`CreateBy`、`CreateTime`、`ModifyBy`、`Version` 等）
- `FillModel`：对象覆盖
- `Validate(Context)`：基于数据注解验证

## 8.2 字段特性

- `[Column]`：表格列元数据（排序、查询、宽度、固定列等）
- `[Form]`：表单字段元数据（布局、只读、占位符等）

## 8.3 数据库访问

使用 `Database` 轻量 ORM：

- `QueryPageAsync<T>` 分页
- `QueryByIdAsync<T>` 查询
- `SaveAsync(entity)` 保存
- `DeleteAsync<T>(id)` 删除
- `TransactionAsync(...)` 事务封装

数据库通过 `option.App.Database = db => db.AddSQLite/...` 注入。

---

## 9. 权限与菜单体系

权限由以下元数据协同完成：

- 页面级：`[Menu]`、`[Route]`
- 操作级：`[Action]`
- 角色页签：`[Role]`、`[TabRole]`

在 `WorkList` 中的 `[Action(Tabs=[...])]` 可做到“按业务状态显示动作”。

---

## 10. 插件体系（页面区块/导航/开发中心）

核心类型：

- `IPlugin`、`PluginBase<T>`
- `PluginAttribute` 家族：
  - `DevPluginAttribute`
  - `NavPluginAttribute`
  - `PagePluginAttribute`
- `PluginConfig`：插件索引中心
- `PluginPage`：页面插件容器

能力：

- 页面可视化添加区块
- 区块参数配置持久化（`Info.Setting` JSON）
- 支持布局类型（列布局、自定义布局）

## 11. 工作流、导入、任务扩展

## 11.1 工作流

示例：`WorkFlows/WorkFlow.cs` 继承 `FlowBase`。

- 定义流程编码、名称
- 在业务创建时调用 `CreateFlowAsync`

## 11.2 导入

示例：`WeatherImport.cs`

- `[Import(typeof(TargetType))]`
- 继承 `ImportBase<T>`
- 在 `InitColumns` 定义可导入字段

## 11.3 后台任务

通过 `[Task("BizType")]` 绑定任务类型，框架启动时自动注册。

## 12. 前端资源与样式组织

当前全局样式主文件：`Known/wwwroot/css/web.css`。

加载路径：

- 框架默认在 `Extension.AddKnown` 中注入：
  - `_content/Known/css/web.css`
  - `_content/Known/js/web.js` 等
- 业务插件可追加：
  - `KStyleSheet.AddStyle("_content/YourPlugin/css/web.css")`

建议：

- 样式变量统一走 `:root` 自定义属性
- 深色主题通过 `:root[data-theme=dark]` 覆盖
- 业务样式避免污染全局命名，统一模块前缀

## 13. 新增业务模块的标准步骤（实战模板）

1. 在插件项目新增实体（继承 `EntityBase`）
2. 新增服务接口 + `[Client]` + `[WebApi, Service]`
3. 新增页面 `BaseTablePage<T>` + `[Route]` + `[Menu]`
4. 给页面动作方法加 `[Action]`
5. 在 `AppModule.AddSample`（或你的模块入口）添加一级菜单
6. 在宿主 `AddApplication` 确认 `services.AddYourModule()` 已调用
7. 启动后检查：菜单、权限点、API、数据表是否生效

---

## 14. 多端支持策略（Server / Wasm / Desktop / MAUI）

推荐做法：

- 业务逻辑尽量放在服务层与实体层
- UI 层优先复用 `Known` 组件模型
- 平台差异放在宿主工程（如 `Program.cs` / `AppConfig.cs`）
- 移动/桌面限制能力（如文件系统、通知方式）通过配置开关隔离

## 15. 常见问题与排查路径

### 15.1 菜单不显示

排查：

- 页面是否有 `[Route]` + `[Menu]`
- 模块程序集是否 `Config.AddModule(...)`
- 是否在宿主执行了 `services.AddSample()`（或自定义模块注入）

### 15.2 动态 API 调不通

排查：

- 服务类是否加 `[WebApi, Service]`
- 是否走了 `services.AddKnownWeb(...)`
- Wasm 客户端 `BaseAddress` 是否正确指向 `api`

### 15.3 表不存在

排查：

- 实体是否继承 `EntityBase`
- 安装/迁移是否执行（`InstallService`、`AppMigrate`）
- 连接串配置是否正确

### 15.4 权限动作缺失

排查：

- 方法是否加 `[Action]`
- 当前角色是否赋权
- `Tabs` 条件是否导致隐藏

---

## 16. 推荐开发规范

- **一个业务能力 = 实体 + 服务 + 页面 + 菜单元数据**
- 服务接口优先，客户端/服务端都实现接口
- 尽量使用 `TransactionAsync` 包裹写操作
- 统一使用 `Result` 返回业务结果
- 公共文案放多语言，不写死
- 样式按模块命名空间隔离，减少全局冲突
- 配置与扩展点集中在 `AppConfig`/`CoreConfig`/`UIConfig`
